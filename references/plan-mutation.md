# Plan Mutation Protocol — 计划突变协议

> 借鉴 OMO SPM 的 plan-mutation 机制。WBS 计划确认后仍可变更，但必须留痕。

## 铁律

1. 所有突变必须在 WBS 台账的 **Mutation Log** 中记录
2. 突变涉及依赖关系变化时，必须验证无循环依赖
3. 计划审查（如有）通过后的突变，需要**重新触发审查**（如果是大突变）
4. 突变不删除原行——使用 `skipped` 状态 + 原因

---

## 5 种突变操作

### 1. Split — 拆分

**触发条件**: 任务太大，一个 subagent 无法在合理时间内完成（> 30 分钟或 > 5 个文件）

**操作**:
```
原任务 ID: 3 → 标记 skipped + 原因"拆分为 3.1 / 3.2"
新任务: 3.1, 3.2 — 继承原任务依赖，3.2 depends on 3.1
```

**Mutation Log 记录**:

| 时间 | 变更类型 | 影响任务 | 原因 | 新任务 |
|------|---------|---------|------|--------|
| 06-12 07:20 | split | 3 | 任务太大，涉及 8 个文件需拆分 | 3.1, 3.2 |

**后续**:
- 3.1 和 3.2 的 Context Brief 必须各自完整
- 原来依赖 Task 3 的任务改为依赖 3.2

**自检**:
- [ ] 原任务状态改为 `skipped`，原因清晰
- [ ] 新任务 ID 使用 X.1, X.2 格式
- [ ] 新任务 Dependencies 已更新（3.2 depends on 3.1）
- [ ] 下游任务的 Dependencies 已更新（从 3 → 3.2）

---

### 2. Insert — 插入

**触发条件**: 执行中发现遗漏了必要的前置工作（如"需要先加环境变量"）

**操作**:
```
在 Task N 和 Task N+1 之间插入 Task N.5
```

**Mutation Log 记录**:

| 时间 | 变更类型 | 影响任务 | 原因 | 新任务 |
|------|---------|---------|------|--------|
| 06-12 07:20 | insert | 2→3 | 发现缺少数据库迁移脚本 | 2.5 |

**规则**:
- 插入位置前后任务的 Dependencies 更新
- 被依赖的新任务 ID 必须出现在后续任务的 Dependencies 中

**自检**:
- [ ] 新任务 ID 使用 N.5 格式（插入点前后序号之间的小数）
- [ ] 后续所有任务的 Dependencies 已重新审查
- [ ] 无循环依赖（A→B→A）
- [ ] Context Brief 完整

---

### 3. Skip — 跳过

**触发条件**: 任务已不需要（需求变更、已被其他任务覆盖、前提条件不满足）

**操作**:
```
Task N → 标记 skipped + 原因
```

**Mutation Log 记录**:

| 时间 | 变更类型 | 影响任务 | 原因 | 新任务 |
|------|---------|---------|------|--------|
| 06-12 07:20 | skip | 4 | 需求变更，不再需要搜索功能 | - |

**后续**:
- 被跳过任务的依赖者（dependents）的 Dependencies 列移除被跳过的 ID
- 如果依赖者缺少必要前置，需要插入新任务或重新评估

**自检**:
- [ ] 任务状态为 `skipped`，原因明确
- [ ] 下游任务的 Dependencies 已更新（移除该 ID）
- [ ] 下游任务无悬空依赖

---

### 4. Reorder — 重排

**触发条件**: 发现依赖关系有误，或并行机会被遗漏

**操作**:
```
更新任务 ID 顺序 + Dependencies 列
```

**Mutation Log 记录**:

| 时间 | 变更类型 | 影响任务 | 原因 | 新任务 |
|------|---------|---------|------|--------|
| 06-12 07:20 | reorder | 5,6 | Task 5 和 6 无文件冲突，可并行 | - |

**规则**:
- 重排后必须运行依赖验证（检查循环依赖、悬空依赖）
- 并行化时，在 Dependencies 列去掉不必要的依赖

**自检**:
- [ ] 所有任务的 Dependencies 引用有效
- [ ] 无循环依赖
- [ ] 并行任务确实无共享状态

---

### 5. Abandon — 废弃

**触发条件**: 方向完全错误（如"不应该用 Redis，应该用 PostgreSQL"），不只是跳过

**操作**:
```
Task N → 标记 skipped + 原因"废弃：具体原因"
```

**Mutation Log 记录**:

| 时间 | 变更类型 | 影响任务 | 原因 | 新任务 |
|------|---------|---------|------|--------|
| 06-12 07:20 | abandon | 6 | 废弃：Redis 方案改为 PostgreSQL，需新建 Task | 7(新建) |

**规则**:
- 必须注明废弃原因和替代方案
- 废弃的任务在 Delivery Summary 中列出为"已废弃"
- 如果引入替代任务，走 Insert 流程

#### Abandon 回滚指引

如果废弃了原方案、切换到新方案后失败，按以下步骤回退：

```
1. 停止当前执行
2. 检查 Mutation Log → 找到 abandon 记录 → 获取原任务 ID
3. 原任务从 skipped 改回 todo（恢复状态）
4. 新方案任务标记 skipped + 原因"回退：新方案失败，恢复原方案"
5. Mutation Log 记录回退操作
6. 更新所有受影响任务的 Dependencies
7. 重新验证依赖链
8. 继续执行原方案
```

**铁律**: abandon 不是永久删除。原任务永远保留在台账中（skipped 状态），随时可恢复。

## 依赖验证（每次突变后）

突变完成后，必须验证依赖链：

```bash
# 1. 检查所有 Dependencies 列引用的 ID 存在
grep -oE '\[[0-9]+(\.[0-9]+)?\]' WBS.md | sort -u

# 2. 检查无循环依赖（简化版）
# 构建依赖图：A → [B, C]
# 对每个任务，DFS 检查是否能回到自身

# 3. 检查无悬空依赖（引用了 skipped/abandoned 的任务）
# 对每个 doing/todo 任务，其所有 dep 必须状态为 done
```

如果验证失败 → 修正 WBS 或触发新突变。

---

## 与 WBS 台账的关系

Mutation Log 是 WBS 台账的**审计轨迹**。任何时候查看 WBS，都必须能追溯计划的演变历史。

**禁止**:
- ❌ 删除 Mutation Log 行
- ❌ 修改旧记录（错误时追加更正说明）
- ❌ 隐藏失败的突变（失败的也记录）

**鼓励**:
- ✅ 详细原因说明
- ✅ 关联的具体错误或决策点
- ✅ 多轮突变时保持连续记录

---

## 何时触发重新审查

| 突变类型 | 是否需重新审查 |
|---------|--------------|
| skip / reorder | 否（轻量） |
| split / insert | 是（大突变，引入新任务） |
| abandon | 是（方向变更） |

重新审查：重新触发 `workflows/writing-plans.md` 中的 adversarial plan review 流程。
